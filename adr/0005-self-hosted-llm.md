# ADR-0005: Разместить LLM-ядро self-hosted на GPU

## Статус

Принято

## Дата

2026-06-29

## Контекст

ИИ-диагностика (RAG + LLM) - ядро ценности системы и предмет дипломной работы. В презентации ИИ-ядро показано как «сервер с GPU». База знаний и разобранные кейсы содержат внутренние регламенты и историю обслуживания объектов ВСМ; передача этих данных во внешний LLM-API спорна по требованиям информационной безопасности. RAG (поиск по базе знаний) остаётся внутренним сервисом в любом варианте; открытым остаётся вопрос размещения генеративной модели.

## Решение

LLM разворачивается self-hosted на GPU-сервере в доверенной серверной зоне (on-prem или частное облако). Модель подключается к Search/RAG Service через адаптер, чтобы её можно было заменить без изменения остальной системы. Внешние облачные LLM-API в MVP не используются.

## Альтернативы

| Альтернатива | Плюсы | Минусы |
|---|---|---|
| Внешний LLM-API (GigaChat, YandexGPT и др.) | Проще и дешевле в эксплуатации | Данные уходят провайдеру, риски ИБ |
| Self-hosted LLM на GPU | Данные не покидают доверенный контур, контроль над моделью | Нужна GPU-инфраструктура и её эксплуатация |
| Гибрид через адаптер | Гибкость выбора модели | Больше работы, риск утечки при включении внешнего провайдера |

## Последствия

Положительные:

- Данные базы знаний и кейсов не покидают доверенную зону.
- Контроль над версией модели и стоимостью инференса.
- Адаптер позволяет заменить модель без переработки системы.

Отрицательные:

- Нужна GPU-инфраструктура (капитальные затраты и эксплуатация).
- Качество может уступать топовым облачным моделям.

## Когда пересмотреть

- Если появится разрешённый по ИБ внешний LLM с приемлемым качеством и стоимостью.
- Если эксплуатация GPU окажется неоправданной при низкой реальной нагрузке.
