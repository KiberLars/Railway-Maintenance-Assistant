# ADR-0007: Ввести контур обучения на разобранных случаях

## Статус

Принято

## Дата

2026-06-29

## Контекст

В концепции заявлен «контур обучения»: каждый разобранный случай должен пополнять базу знаний и улучшать будущие подсказки. Нужно превращать полевой опыт в воспроизводимое знание, но без риска занесения в контент «фантазий» модели или непроверенных решений.

## Решение

Вводится Learning/Feedback Service. По итогам осмотра (особенно после правок эксперта) формируется кандидат в базу знаний. Кандидат проходит проверку эксперта или администратора и только после этого публикуется через Documentation Service как часть новой `knowledge_base_version`. Автоматическое дообучение LLM в MVP не выполняется: «обучение» реализуется как обогащение корпуса базы знаний для RAG, а не как переобучение модели.

## Альтернативы

| Альтернатива | Плюсы | Минусы |
|---|---|---|
| Без контура обучения | Проще | Опыт теряется, подсказки не улучшаются |
| Автоматическое дообучение LLM на кейсах | Максимальная адаптация | Дорого, риск дрейфа и утечки, сложный контроль качества |
| Кандидаты в базу знаний с проверкой человеком | Знания растут под контролем качества, без галлюцинаций | Нужна модерация и процесс публикации |

## Последствия

Положительные:

- База знаний растёт на реальном опыте, качество подсказок RAG повышается.
- Человеческая проверка не допускает ошибочный контент в публикацию.

Отрицательные:

- Нужен процесс и роль для модерации кандидатов.
- Эффект проявляется постепенно, по мере накопления кейсов.

## Когда пересмотреть

- Если объём кейсов потребует частичной автоматизации модерации.
- Если появятся потребность и ресурсы для дообучения собственной модели.
