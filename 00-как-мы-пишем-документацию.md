# 00. Как мы пишем документацию

## Цель

Этот раздел фиксирует правила работы над архитектурной документацией RMA. Документация должна быть не разовым отчётом, а стартовым набором материалов для обсуждения MVP, проверки решений и дальнейшей разработки.

## Процесс

1. Сначала фиксируются границы системы, пользователи, сценарии и требования.
2. Затем требования связываются с архитектурными драйверами.
3. После этого описываются контейнеры, компоненты, данные, сценарии и развёртывание.
4. Все решения, влияющие на архитектуру, надёжность, безопасность или развитие, оформляются в ADR.
5. После каждого крупного изменения проверяется согласованность терминов, диаграмм и требований.

## Контекст для LLM-агента

При работе с LLM-агентом в контекст передаются:

- название и назначение системы: `RMA` - интеллектуальная поддержка ТО объектов ВСМ;
- граница MVP: Android-планшеты и смартфоны;
- роли: технический специалист, диспетчер, эксперт, администратор;
- логические контуры: полевой, данные и знания, ИИ, операционно-экспертный;
- ключевое решение: полная база знаний кэшируется локально;
- ключевое решение: OCR локальный; LLM/RAG, STT/TTS и видеосвязь только онлайн; LLM self-hosted на GPU;
- ключевое решение: сложные случаи решает эксперт по видео, разобранные случаи пополняют базу знаний;
- список разделов документации и ADR;
- открытые допущения.

## Правила качества

- Не добавлять новые компоненты без связи с требованиями.
- Не смешивать MVP и будущее развитие.
- EAM не входит в проект: не описывать её как зависимость, интеграцию или future scope.
- Не писать "система работает офлайн" без уточнения: офлайн доступны локальная база знаний, OCR, чек-листы и журналы; ИИ-функции и видеосвязь требуют сети.
- Диаграммы должны объяснять архитектуру, а не дублировать таблицы.
- Диаграммы выдержаны в едином стиле: flowchart с прямыми (90°) связями для структуры, sequence - для потоков.
- Все спорные места фиксируются как допущения.

## Проверочный список

- Термины `RMA`, `Мобильное приложение`, `база знаний`, `MaintenanceRequest`, `MaintenanceJob`, `OperationLog`, `ExpertConsultation`, `KnowledgeCandidate` используются одинаково.
- Для каждого сложного решения есть ADR или ссылка на него.
- Диаграммы соответствуют тексту.
- Ошибочные и офлайн-сценарии описаны явно.
- Недоказанные детали не выданы за факты.
